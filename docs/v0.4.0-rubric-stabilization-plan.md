## 1. Release objective

Version `0.4.0` converts the 16 ambiguity classes recorded in `HIT-ARFR-001` into a synchronized, testable assessment contract.

The release is a normative-method update. It does not establish human inter-rater reliability, field effectiveness, legal compliance, certification, or independent adoption.

## 2. Approved contract changes

ADR-0002 approves:

- an evidence-state model separating affirmative absence, formal presence, operational capability, and observed exercise;
- explicit thresholds for `0`, `1`, `2`, and `IE`;
- dimension-specific rules for Counsel, Judgment, Command, Correction, Repair, and Reform;
- Repair trigger states;
- separate institutional-record and assessment-packet integrity statuses;
- deterministic derivation of the overall Telemetry Integrity status;
- a dominant-pattern default for period-level aggregation;
- a required actor-authority matrix;
- source-conflict weighting rules;
- construct-specific cross-dimensional evidence reuse;
- precise citation locators and bounded evidence propositions.

ADR-0003 preserves chronological semantic versioning. The earlier planned `v0.3.0` result label is superseded; the locked human result must use the next available version when completed.

## 10. Canonical-promotion requirements

A dedicated release pull request must:

1. promote the candidate specification, schema, catalog, and handbook to canonical paths;
2. update README, ROADMAP, RESEARCH, LIMITATIONS, PROVENANCE, CHANGELOG, CITATION.cff, `.zenodo.json`, and release notes;
3. remove current documentation that treats `v0.3.0` as a future chronological release;
4. preserve all historical case files and migration dispositions;
5. point release validators to canonical artifacts;
6. pass the adjacent-system claim audit;
7. pass all validation on the exact release commit.

## 11. Stop conditions

Do not tag `v0.4.0` while:

- candidate files remain noncanonical;
- release metadata remains `0.2.1`;
- any current document promises a later `v0.3.0` release;
- historical and migrated case records are not clearly separated;
- the locked protocol or historical findings have changed;
- CI has not passed on the exact promotion commit.
