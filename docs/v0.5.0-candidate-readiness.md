# HIT v0.5.0 Candidate Readiness Record

**Status:** Implementation candidate complete; release promotion pending  
**Date:** 2026-07-18  
**Base release:** `v0.4.0`  
**Target repository release:** `v0.5.0`  
**Normative specification, schema, and catalog:** `0.4.0`  
**Research maturity:** Level 1, Defined

## 1. Advancement claim

The candidate implements executable conformance for complete HIT assessment records. It does not revise the normative `0.4.0` method.

The implementation can:

- validate one complete `0.4.0` assessment structurally and semantically;
- identify cross-record actor, evidence-claim, finding, integrity, Repair, aggregation, and citation defects through stable error codes;
- run the released `0.4.0` validation and the new complete-record suite through one command;
- produce deterministic text and JSON reports;
- produce non-mutating reconstruction plans for historical `0.1.0` assessments;
- prove that all four historical public assessments remain byte-for-byte unchanged during migration planning.

## 2. Completed work packages

### WP-A: Complete-assessment engine — complete

The engine checks:

- released JSON Schema conformance;
- exact six-dimension coverage;
- actor and claim identifier uniqueness;
- actor, related-actor, and evidence-claim references;
- actor-to-claim attribution reciprocity;
- evidence relation and dimension consistency;
- finding-to-evidence-state consistency;
- support for determinate findings;
- unresolved propositions and search records for `IE`;
- Repair-trigger consistency;
- Telemetry Integrity derivation;
- assessment-scope and aggregation consistency;
- reproducibility notes for nonstandard locators.

### WP-B: Deterministic conformance suite — complete

The suite contains sixteen complete-record cases derived from the canonical `0.4.0` assessment:

- one valid record;
- fifteen invalid records covering the implemented cross-record rules;
- expected error-code assertions;
- stable case and issue ordering;
- canonical JSON SHA-256 digests;
- two-run byte-stability verification.

The existing 48 isolated rubric-boundary fixtures remain part of release validation and are not replaced by this suite.

### WP-C: Compatibility and migration — complete

The machine-readable compatibility manifest declares:

- full conformance support for specification, schema, and catalog `0.4.0`;
- historical-validation-only support for contract `0.1.0`;
- no automatic `0.1.0` to `0.4.0` conversion;
- mandatory preservation of the original record;
- fresh reassessment as the only supported migration method.

All four historical assessment files are hashed before and after migration planning. The self-test fails if any file changes.

### WP-D: Public operation — complete

The public package entry point is:

```bash
python -m src conformance --all
python -m src conformance --path assessment.json
python -m src migration-plan --path historical-assessment.json
```

Argument parsing, execution, commands, configuration, models, validation, and conformance orchestration are separated by function under `src/`.

The CLI supports text and JSON output, optional output files, deterministic report ordering, and nonzero failure exit codes.

### WP-E: Release promotion — pending by design

A separate release-promotion pull request must:

- advance repository metadata to `0.5.0`;
- keep specification, schema, and catalog at `0.4.0`;
- update the changelog, roadmap, provenance, citation metadata, Zenodo metadata, and release notes;
- convert development engine labels to released `0.5.0` labels;
- extend exact-release validation to the implementation package;
- pass exact-head CI before merge;
- tag the exact verified merge commit as `v0.5.0`.

## 3. Validation evidence

The candidate workflow validates:

1. deterministic historical fixture formatting;
2. the complete released `0.4.0` contract;
3. the new repository-wide executable-conformance command;
4. individual valid and invalid CLI behavior;
5. deterministic report output;
6. protected migration planning;
7. publication of machine-readable validation reports as a workflow artifact.

The implementation head before this readiness record passed HIT Validation run 66. The final pull-request head must pass the same workflow before review or merge.

## 4. Protected research and governance boundaries

The candidate does not:

- modify protocol `HIT-IRP-CIGNA-001` or its frozen `0.1.0` scorer contract;
- modify the scorer threshold, critical-disagreement rule, or publication obligation;
- create a human inter-rater result;
- advance H3 or Maturity Level 2;
- overwrite or re-score a historical public assessment;
- create a `0.4.0` Cigna assessment before the locked result is published;
- change a substantive HIT finding;
- claim legal compliance, standards conformity, certification, empirical validity, runtime enforcement, signed-receipt verification, policy-pack harmonization, or evidence portability.

## 5. Stop conditions

The candidate must not be merged when any of the following is true:

- exact-head CI is incomplete or failing;
- a complete-record vector does not produce its declared result;
- repeated reports over identical inputs differ;
- migration planning changes a historical source file;
- the compatibility manifest permits automatic migration;
- the `0.4.0` release validator regresses;
- current release metadata is advanced before the separate release-promotion review;
- a protected human-protocol or maturity claim changes.

## 6. Disposition

The implementation candidate is ready for maintainer review once its final exact head passes CI. Merge approval applies only to the implementation candidate. It does not authorize publication of `v0.5.0`; release publication remains a separate, reviewable decision.
