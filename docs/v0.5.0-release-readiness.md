# HIT v0.5.0 Release Readiness

**Status:** Promotion candidate  
**Repository target:** `0.5.0`  
**Normative contract:** specification, schema, catalog, and handbook `0.4.0`  
**Research maturity:** Level 1, Defined

## Release proposition

Version `0.5.0` publishes the executable conformance implementation merged through PR #11. It does not revise the normative assessment contract.

## Required promotion state

- repository metadata reports `0.5.0`;
- conformance engine and compatibility manifest report `0.5.0`;
- specification, assessment schema, dimension catalog, and handbook remain `0.4.0`;
- all four historical `0.1.0` assessments remain byte-identical;
- automatic migration remains prohibited;
- protocol `HIT-IRP-CIGNA-001` and its frozen packet remain unchanged;
- H3 remains unresolved and maturity remains Level 1;
- no standalone software DOI is claimed before successful archival;
- exact-head CI passes before merge;
- tagging and release publication require separate maintainer approval.

## Validation surface

The exact promotion head must pass:

1. deterministic fixture formatting;
2. release metadata and contract-preservation validation;
3. all 48 rubric boundary fixtures;
4. all 16 complete-record conformance vectors;
5. compatibility-manifest validation;
6. migration preservation for all four historical assessments;
7. public CLI smoke tests;
8. byte-stable report generation;
9. validation-report artifact upload.

## Stop conditions

Do not merge when any of the following is true:

- a normative `0.4.0` artifact changed without a separately reviewed contract defect;
- a historical assessment blob changed;
- the locked human protocol changed;
- a development engine marker remains in a current public artifact;
- release metadata conflicts across README, CFF, Zenodo, changelog, provenance, compatibility manifest, or release notes;
- any release or conformance validation fails;
- the branch moved after the validated run.

## Publication boundary

Merging the promotion pull request does not itself create tag `v0.5.0` or publish a GitHub release. The exact merge commit must be verified first. Tag and release publication remain separate actions.
