# HIT v0.5.0 Executable Assessment Conformance Plan

**Status:** Active development plan  
**Base release:** `v0.4.0`  
**Target repository release:** `v0.5.0`  
**Normative contract:** specification, schema, and catalog `0.4.0` unless an independently reviewed defect requires a separate change decision  
**Research maturity:** remains Level 1

## 1. Purpose

Version `0.4.0` stabilized the rubric and converted all sixteen known ambiguity classes into executable boundary rules. Version `0.5.0` must move from isolated rule vectors to complete-record conformance without silently changing the normative method.

The central test is whether a third party can submit one full HIT assessment and receive a deterministic report identifying structural and cross-record defects.

## 2. Architectural direction

The implementation follows the repository structure preferred for future software work:

```text
src/
  cli.py                         -- CLI entry point
  commands/                      -- conformance and migration commands
  config/                        -- repository paths and version configuration
  models/                        -- typed conformance result objects
  validation/                    -- complete-assessment semantic validation
  conformance/                   -- suite, compatibility, migration, and report orchestration
  rubric/                        -- existing 0.4.0 boundary-rule evaluators
compatibility/                   -- machine-readable contract support manifest
fixtures/v0.5.0-conformance/     -- complete-record valid and invalid vectors
scripts/                         -- backward-compatible repository wrappers
```

Existing scripts remain supported until the CLI has equivalent or stronger coverage. The refactor must not weaken the locked protocol, historical case preservation, or release checks.

## 3. Work packages

### WP-A: Complete-assessment engine

- JSON Schema validation against the released `0.4.0` contract;
- exact dimension coverage;
- actor and claim identifier uniqueness;
- actor, related-actor, and claim-reference integrity;
- actor-to-claim attribution reciprocity;
- evidence relation and dimension consistency;
- finding-to-evidence-state consistency;
- support requirements for determinate findings;
- `IE` proposition and search requirements;
- Repair trigger consistency;
- split Telemetry Integrity derivation;
- assessment-scope and aggregation consistency;
- precision requirement for `other` locators.

### WP-B: Deterministic conformance suite

- one canonical valid record;
- derived invalid complete records for every implemented cross-record rule;
- deterministic mutation application;
- expected error-code assertions;
- stable case ordering;
- SHA-256 digests for derived records;
- no timestamps in reproducibility reports.

### WP-C: Compatibility and migration

- machine-readable compatibility manifest;
- full support for contract `0.4.0`;
- historical-validation-only support for `0.1.0`;
- explicit prohibition on automatic `0.1.0` to `0.4.0` conversion;
- non-mutating migration plans requiring fresh reassessment and original preservation.

### WP-D: One-command operation

```bash
python -m src.cli conformance --all
python -m src.cli conformance --path assessment.json
python -m src.cli migration-plan --path historical-assessment.json
```

Reports must support deterministic text and JSON output and return a nonzero exit code on failure.

### WP-E: Release promotion

- CI executes the one-command conformance suite;
- README project structure and usage are synchronized;
- release and compatibility notes distinguish repository `0.5.0` from the unchanged normative `0.4.0` contract;
- exact-head CI passes before merge;
- exact merge commit is tagged only after post-merge verification.

## 4. Protected boundaries

This workstream must not:

- revise protocol `HIT-IRP-CIGNA-001` or its frozen `0.1.0` scorer contract;
- create or imply a human reliability result;
- advance H3 or Maturity Level 2;
- re-score or overwrite the four historical public assessments;
- create a `0.4.0` Cigna assessment before the locked result is published;
- claim legal, standards, certification, runtime-enforcement, receipt, control-harmonization, or evidence-portability capability.

## 5. Completion test

Version `0.5.0` is ready only when the public implementation rejects every included invalid complete-record state, accepts every included valid state, preserves historical records, produces byte-stable JSON reports for identical inputs, and requires no private author explanation to run.
